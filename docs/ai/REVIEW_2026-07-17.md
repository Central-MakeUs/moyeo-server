# Code Review Report - 2026-07-17 (Meeting Cover Image)

## Scope

- Optional meeting-cover upload as multipart form data
- Server-side image resize and private S3 storage
- Host cover replacement/deletion and invite-link image retrieval
- Cache-versioned cover URL, error mapping, and rollback/commit cleanup

## Initial Findings and Applied Resolution

| Severity | Category | Finding | Resolution |
| --- | --- | --- | --- |
| MEDIUM | CONCRETE_RISK | A stable cover-image URL combined with `immutable` cache control could keep a replaced image in browser cache for up to one year. | Return `coverImageUrl` with a version derived from the current S3 object key. No additional DB column is used. |
| MEDIUM | CONCRETE_RISK | S3 missing-object and unavailable-storage failures were mapped to common, misleading errors. | Add dedicated `MEETING_COVER_IMAGE_NOT_FOUND` (404) and `MEETING_COVER_IMAGE_UNAVAILABLE` (503) errors. |
| MEDIUM | CONCRETE_RISK | Uploading/deleting S3 objects inside the database transaction could leave incorrect object state after a DB rollback. | Delete a newly uploaded object on transaction rollback; delete a replaced/deleted former object only after commit. |
| MEDIUM | DOCUMENTATION_MISMATCH | The policy allowed host deletion, but there was no delete API. | Add authenticated host-only `DELETE /api/meetings/{meetingId}/cover-image`. |
| MEDIUM | TEST_GAP | The initial implementation had no multipart cover flow test. | Add integration tests for multipart creation and host replacement/deletion. |

## Follow-up Review Findings

### Finding 1

Severity: MEDIUM

Category: CONCRETE_RISK

File: `src/main/java/com/moyeo/service/meeting/MeetingCoverProcessor.java`

Location: `resizeToJpeg`, lines 29-49

Policy: No explicit pixel-count limit is documented; `DOMAIN-001` requires the
technical validation behavior to remain explicit.

Evidence: The processor limits encoded upload bytes to 5 MB, then calls
`ImageIO.read` before checking image dimensions. A small, highly compressed PNG
can decode into a very large `BufferedImage`.

Concrete Risk: An authenticated client can exhaust application heap during image
decode and make the current API unavailable. The existing 1280x720 output bound
does not prevent the input image from first being decoded at its original size.

Confidence: HIGH

Recommended direction: Add a temporary, configuration-backed maximum input
pixel count or width/height bound before full decode, document it as a technical
MVP setting, and return a stable validation error when exceeded.

Human Decision: DEFERRED - Record as a temporary MVP follow-up TODO in
`docs/policies/MEETING_PARTICIPATION_POLICY.md`.

### Finding 2

Severity: MEDIUM

Category: TEST_GAP

File: `src/test/java/com/moyeo/controller/meeting/MeetingControllerTest.java`

Location: meeting-cover test cases

Policy: `docs/ai/CODE_REVIEW.md` requires verification for high-risk changed
behavior.

Evidence: Current tests cover successful multipart creation and successful
host replacement/deletion. They do not cover a non-host modification attempt,
invalid/oversized upload rejection, storage-unavailable response, or rollback
cleanup after an upload followed by a transaction failure.

Concrete Risk: The new authorization, validation, and error-contract branches
can regress without failing the current test suite.

Confidence: HIGH

Recommended direction: Add focused tests for 403, 413/415, 503, and rollback
compensation behavior.

Human Decision: DEFERRED - Record as a temporary MVP follow-up TODO in
`docs/policies/MEETING_PARTICIPATION_POLICY.md`.
